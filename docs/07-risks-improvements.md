# 风险与改进建议小结

关联文档：`00-project-overview.md`、`01-api-reference.md`、`02-boost-asio-beast.md`、`03-grpc-distributed.md`。

## 1. 可扩展性风险
- **StatusServer token 内存存储**：单实例内存 map，重启即失效。
  - **改进**：引入 Redis/Etcd 等分布式存储，或 JWT 无状态方案。
- **消息处理单线程队列**：ChatServer `LogicSystem` 单消费者。
  - **改进**：多队列 + 多消费者，按会话/用户分区。

## 2. 安全性风险
- **HTTP/gRPC/TCP 无 TLS**：所有链路明文传输。
  - **改进**：HTTPS + gRPC TLS + TCP TLS（或内网隔离 + mTLS）。
- **验证码/密码策略**：验证码明文存 Redis，密码哈希未加盐。
  - **改进**：采用加盐哈希（bcrypt/argon2），验证码限流。

## 3. 稳定性与一致性风险
- **幂等与重试**：请求级幂等处理不足，重试可能重复写入。
  - **改进**：引入幂等键、事务化操作。
- **离线消息与投递保证**：当前只写库并即时通知。
  - **改进**：引入消息队列，支持 ACK/重试/离线投递。

## 4. 运维与可观测性风险
- **缺少监控与追踪**：无统一 metrics/tracing。
  - **改进**：Prometheus + Grafana + OpenTelemetry。
- **配置与密钥管理**：配置中包含敏感信息。
  - **改进**：使用配置中心与密钥管理服务。

## 5. 代码层面改进建议
- **消息 ID 缺失定义**：部分 TCP 消息 ID 在代码中使用但未定义。
  - **改进**：统一枚举定义与协议文档同步。
- **错误码扩展**：业务错误码与系统错误码混合。
  - **改进**：分层错误码或结构化错误返回。

---

## 面试总结口径
- 现有架构完成了“可用的分布式 IM 核心链路”。
- 下一步改造重点：**安全（TLS）、一致性（幂等）、可观测性（监控/日志/追踪）、高可用（token/状态分布式）**。
